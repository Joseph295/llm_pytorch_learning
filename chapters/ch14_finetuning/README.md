## 14.1 来龙去脉：为什么预训练之后还要微调

第 9 章你训出的模型只会"续写"——它学到了语言的统计规律，但不会"听指令"。给它 "把这段话翻译成英文"，它可能续写出更多类似的中文指令而不是执行。这是预训练模型的本质：**它建模的是 P(下一个词 | 前文)，不是 P(有用的回答 | 指令)**。

从"会续写"到"能干活"的鸿沟，靠微调三阶段填平（现代 LLM 的标准流程）：

1. **预训练（Pretraining）**：海量无标注文本上学语言和世界知识（第 9-13 章）。最贵，通常只有大厂做。产出"基座模型"（base model）。
2. **监督微调（SFT, Supervised Fine-Tuning）**：在"指令-回答"对上继续训练，教模型**遵循指令的格式和行为**。数据量小（几千到几十万条），大多数人的实战起点。产出"指令模型"（instruct model）。
3. **偏好对齐（RLHF / DPO）**：用人类偏好数据（"回答A比回答B好"）进一步调整，让输出更符合人类期望（有用、无害、诚实）。产出"对齐模型"。

**核心矛盾**：全参数微调一个 7B 模型要 112GB 显存（第 2 章，和预训练一样贵）——大多数人负担不起。**参数高效微调（PEFT）**就是为解决这个矛盾而生：只训练极少量参数，达到接近全参微调的效果。LoRA 是其中最成功的方法。这一章的重心，就是理解"为什么少量参数就够"。

## 14.2 核心原理

### ① SFT：换了数据的预训练

SFT 在机制上和预训练几乎一样（还是 next-token prediction + 交叉熵，第 6/9 章的循环），区别只在**数据形态**和**loss 掩码**：

**数据形态**：指令-回答对，用模板拼成一条序列：
```
<|user|>把"你好"翻译成英文<|assistant|>Hello<|end|>
```

**loss 掩码（关键）**：只对"回答"部分算 loss，"指令"部分的 token 用 `ignore_index=-100`（第 7 章 cross_entropy 的 ignore）屏蔽——我们要模型学会"生成回答"，不是"生成指令"。忘了掩码是易错点①（模型会学着复读指令）。

**其他 SFT 特点**：学习率比预训练小一个量级（1e-5 ~ 2e-5，别破坏预训练学到的知识）、epoch 少（1-3，多了过拟合）、常用 packing（第 5 章）或按长度分桶。

### ② LoRA：低秩假设的胜利

**核心洞察**：微调对权重的改变 ΔW 是**低秩**的——适配一个任务不需要改变整个高维权重空间，改变发生在一个低维子空间里。数学上：

```
微调后权重 = W₀ + ΔW，  LoRA 假设 ΔW = B·A  （低秩分解）
其中 W₀ 是 d×d 冻结的预训练权重
     A 是 r×d，B 是 d×r，秩 r << d（典型 r=8/16/64）
```

前向变成 `h = W₀·x + B·A·x`（原路径 + 低秩旁路）。**只训练 A、B**（冻结 W₀），参数量从 d² 降到 2rd。d=4096、r=16 时：全参 1600 万 → LoRA 13 万，**省 99.2%**。

三个精妙之处：
1. **W₀ 冻结不产生梯度和优化器状态**——省的不只是可训练参数，更是第 2 章的梯度（2 字节/参数）和优化器状态（12 字节/参数）大头。这才是 LoRA 省显存的关键。
2. **初始化**：A 用高斯随机、B 初始化为 0 → 训练开始时 ΔW = B·A = 0，模型 = 原模型（不破坏预训练）。然后 B 逐渐学出适配。
3. **缩放因子 α/r**：LoRA 输出乘 α/r 控制适配强度，α 通常设为 r 或 2r。
4. **推理零开销**：部署时可以把 B·A **合并**进 W₀（`W = W₀ + BA`），推理时和原模型完全一样快——不像 adapter 会增加推理延迟。

本章实验从零手写 LoRA 线性层，在小模型上验证"冻结主干 + 训练低秩旁路"能学会新任务，并验证合并后推理等价。

### ③ QLoRA：4-bit 量化 + LoRA，单卡微调 7B

LoRA 省了梯度和优化器状态，但**冻结的 W₀ 本身还占显存**（7B × 2 字节 = 14GB）。QLoRA 再进一步：把冻结的 W₀ 量化到 **4-bit**（14GB → 3.5GB），LoRA 旁路仍用高精度训练。三个关键技术：

1. **NF4（4-bit NormalFloat）量化**：针对正态分布权重优化的 4-bit 格式，比普通 int4 精度损失更小。
2. **双重量化**：连量化的 scale 参数也量化，再省一点。
3. **反量化计算**：前向时把 4-bit 权重临时反量化到 bf16 算矩阵乘（W₀ 只读，量化误差不累积）。

效果：7B 模型 QLoRA 微调从需要 A100-80G 降到能在单张 24GB 卡（甚至你的 M4 统一内存！）上跑。**这是让个人能微调大模型的关键技术**，第 15 章的里程碑就用它。代价：反量化有计算开销，训练比全精度 LoRA 慢一些。

### ④ DPO：把 RLHF 变简单

偏好对齐的经典方法是 **RLHF（基于人类反馈的强化学习）**：训一个奖励模型（打分），再用 PPO（强化学习）优化语言模型最大化奖励。问题是 PPO 复杂、不稳定、要同时加载 4 个模型（策略/参考/奖励/critic），工程噩梦。

**DPO（Direct Preference Optimization）**的洞察：偏好优化可以**跳过显式奖励模型和强化学习**，直接用一个巧妙的分类损失在偏好对上训练。给定"好回答 y_w、坏回答 y_l"，DPO 的损失让模型提高 y_w 相对 y_l 的对数概率比（相对一个冻结的参考模型）：

```
L_DPO = -log σ( β·[log π(y_w)/π_ref(y_w) - log π(y_l)/π_ref(y_l)] )
```

直觉：把"好回答"的概率往上推、"坏回答"往下压，β 控制偏离参考模型的程度（防止跑太远丢失能力）。只需 2 个模型（策略 + 冻结参考）、标准梯度下降、稳定好训——DPO 因此成为开源对齐的主流。本章实验实现 DPO 损失，在玩具偏好数据上验证它确实提高了偏好回答的概率。

**方法谱系**（知道有哪些，第 20 章面试）：RLHF-PPO（经典，最强但最复杂）→ DPO（简化，主流）→ 各种变体（IPO/KTO/ORPO/SimPO，各有取舍）。趋势是"更简单、更稳定、更少的模型"。

## 14.4 易错点清单

**① SFT 忘了掩码指令部分**
→ **现象**：模型学会复读/续写指令，而不是回答。
→ **原因**：对整条序列（含指令）算 loss，模型也在学习生成指令 token。
→ **修正**：指令部分的 label 设 -100（ignore_index），只对回答 + 结束符算 loss。

**② LoRA 的 B 没初始化为 0**
→ **现象**：训练一开始就偏离预训练模型（ΔW ≠ 0），可能破坏已有能力、训练不稳。
→ **修正**：A 随机、B 全 0，保证初始 ΔW=0。这是 LoRA 论文的标准做法。

**③ LoRA 加在错误的层**
→ **现象**：效果不如预期。LoRA 通常加在注意力的 q/v 投影（有时全部 q/k/v/o + FFN）。只加少数层欠拟合，加太多失去省参数的意义。
→ **修正**：从"q/v 投影 + r=8"开始，不够再加层/加秩（14.8 有扫描实验）。

**④ QLoRA 把 LoRA 旁路也量化了**
→ **现象**：训练不动或效果差。量化的是**冻结主干** W₀，LoRA 旁路 A/B 必须保持高精度（它们要接收梯度）。
→ **修正**：确认量化只作用于 base model 的权重，adapter 参数是 bf16/fp32。

**⑤ DPO 的参考模型没冻结 / 忘了 detach**
→ **现象**：训练发散或偏好没学到。参考模型 π_ref 必须冻结（no_grad），只有策略模型 π 训练。
→ **修正**：π_ref 用 `torch.no_grad()` 前向；β 别设太小（跑离参考太远，能力崩坏）也别太大（学不动）。

**⑥ 合并 LoRA 时精度/缩放搞错**
→ **现象**：合并后推理结果和训练时不一致。
→ **修正**：合并公式 `W = W₀ + (α/r)·B·A`，别漏缩放因子；合并在和训练相同的 dtype 下做。

## 14.5 开源项目的最佳实践

**① HF PEFT 库：LoRA 的标准实现**
[huggingface/peft](https://github.com/huggingface/peft)：`LoraConfig(r=16, target_modules=["q_proj","v_proj"], lora_alpha=32)` + `get_peft_model(model, config)` 三行给任意模型加 LoRA。读它的 `LoraLayer` 实现，和你手写的对照——核心逻辑一致，多的是对各种层类型的适配、合并/卸载 API、多 adapter 切换。第 15 章直接用它。

**② TRL 库：SFT/DPO/PPO 的训练器**
[huggingface/trl](https://github.com/huggingface/trl)：`SFTTrainer`（自动处理 packing、掩码模板）、`DPOTrainer`（自动管理参考模型）、`PPOTrainer`。看 `DPOTrainer` 如何构造偏好 batch、算 DPO 损失、管理 π_ref——和你手写的 dpo_loss 对照。它还集成了各种变体（IPO/KTO）。

**③ QLoRA 的 bitsandbytes 后端**
`from transformers import BitsAndBytesConfig; BitsAndBytesConfig(load_in_4bit=True, bnb_4bit_quant_type="nf4", bnb_4bit_compute_dtype=torch.bfloat16)`。这一个配置就是 14.2-③ 的全部（NF4 + 双重量化 + bf16 计算）。读 QLoRA 论文（Dettmers 2023）理解 NF4 的设计。**注意 bitsandbytes 主要支持 CUDA**，M4 上用不了，第 15 章上云才实战。

**④ Unsloth 等优化实现**：把 QLoRA 的反量化 kernel 融合优化，训练快 2 倍、省 40% 显存——生产微调常用。理解原理后，用这些优化过的实现是自然选择。

## 14.6 典型面试题

**Q1：预训练、SFT、RLHF/DPO 三阶段各解决什么问题？数据和目标有何不同？**

> **参考答案**：预训练在海量无标注文本上学语言和知识（目标 next-token，数据 TB 级网页），产出会续写但不听话的 base 模型；SFT 在指令-回答对上教遵循指令（目标仍 next-token 但只对回答算 loss，数据千到十万条高质量对），产出 instruct 模型；RLHF/DPO 用人类偏好数据对齐价值观（目标提高偏好回答的相对概率，数据偏好对），产出对齐模型。三阶段数据量递减、单条价值递增。**加分点**：SFT 决定能力上限，对齐主要调风格/安全；base 模型也可直接用（few-shot）；能力主要来自预训练，微调是"激发"而非"注入"。

**Q2：LoRA 为什么有效？低秩假设的依据？相比全参微调省了什么？**

> **参考答案**：假设微调的权重改变 ΔW 是低秩的（适配任务只需在低维子空间调整），故 ΔW=B·A（r<<d），只训 A/B。依据：实证上低秩适配接近全参效果，且预训练模型的权重更新本身经验上低秩（Aghajanyan 等的内在维度研究）。省的核心不只是可训练参数（d²→2rd），更是冻结主干不产生梯度（2字节/参数）和优化器状态（12字节/参数）——这是显存大头（第 2 章）。**加分点**：B 初始化为 0 保证起点不变；α/r 缩放；推理可合并回主干零开销；缺点是表达力受 r 限制、某些任务不如全参。

**Q3：QLoRA 如何让 7B 微调塞进单张 24GB 卡？三个关键技术？**

> **参考答案**：LoRA 已省掉梯度/优化器状态，QLoRA 再把冻结主干量化到 4-bit（7B 从 14GB 降到 3.5GB）。三技术：NF4（针对正态分布权重的 4-bit 格式，精度损失小）、双重量化（量化 scale 也量化）、反量化计算（前向临时反量化到 bf16 算，主干只读不累积误差）。加上 gradient checkpointing（第 11 章）省激活，7B 微调可在 24GB 内完成。**加分点**：量化误差不影响梯度质量（只在只读的前向路径）；代价是反量化计算开销；bitsandbytes/Unsloth 的 kernel 优化。

**Q4：DPO 相比 RLHF-PPO 简化在哪？DPO 损失的直觉？**

> **参考答案**：PPO 需要训练独立奖励模型 + 用强化学习优化，同时加载 4 个模型（策略/参考/奖励/critic），复杂不稳定；DPO 用数学推导证明可跳过显式奖励和 RL，直接用一个分类损失在偏好对上做标准梯度下降，只需 2 个模型（策略 + 冻结参考）。损失直觉：提高"好回答"相对"坏回答"的对数概率比（相对参考模型），β 控制偏离参考的程度防能力崩坏。**加分点**：DPO 隐式定义了奖励（log 概率比）；β 太大学不动、太小跑偏；变体 IPO（防过拟合）、KTO（无需成对偏好）、SimPO（去掉参考模型）；DPO 对数据质量敏感。

## 14.7 疑难杂症排查

**案例 1：SFT 后模型开始复读指令或胡乱续写**

几乎都是 loss 掩码问题（易错点①）——检查指令部分的 label 是否为 -100。打印一个 batch 的 labels，确认指令 token 位置是 -100、回答 token 位置是真实 id。其次检查对话模板是否和推理时一致（训练用的特殊 token/格式，推理时也要用同样的）。

**案例 2：LoRA 微调 loss 不降**

① LoRA 加的层不对（易错点③）——至少加 q/v 投影；② 学习率太小（LoRA 的 lr 通常比全参微调大，1e-4~3e-4，因为只训少量参数）；③ B 没初始化为 0 导致起点混乱（易错点②）；④ r 太小欠拟合——加大 r 试试；⑤ 忘了把 base model 设 eval/冻结，梯度乱窜。

**案例 3：QLoRA 训练显存仍然爆**

① 序列太长（激活 O(T²)）——减 max_length 或开 gradient checkpointing；② batch 太大——减 batch + 梯度累积（第 6 章）；③ 没真正 4-bit 加载（检查 BitsAndBytesConfig 生效，打印模型看权重 dtype）；④ 优化器状态——用 paged_adamw_8bit（8-bit 优化器，第 11 章 HF 菜单）。第 15 章有完整的"显存不够依次打开"菜单。

## 14.8 练习题

### 基础 1：SFT 掩码
用 `sft_masking.py` 的框架，给一批"指令-回答"对构造正确的 label（指令部分 -100），验证：只对回答算 loss 时，模型学会回答；对全序列算 loss 时，模型复读指令。对比两者的生成行为。

### 基础 2：LoRA 参数量账
对一个 7B 模型（d=4096, L=32），算：只对 q/v 投影加 LoRA(r=16) 的可训练参数量占全参的比例。再算 r=8/32/64/128 各是多少。验证"1% 参数"的说法。

### 进阶 1：手写并验证 LoRA 合并
用 `lora_from_scratch.py`，训练一个 LoRA 层适配小任务，然后把 B·A 合并进 W₀，验证合并后的单一权重矩阵前向输出与"W₀ + LoRA 旁路"完全一致（allclose）。理解为什么 LoRA 推理零开销。

### 挑战 1：完整 DPO 训练
用 `dpo_loss.py` 的损失，在第 9 章的 miniGPT 上做一个玩具 DPO：构造偏好对（如"更长的续写"vs"更短的续写"作为偏好），训练几十步，验证偏好回答的概率相对参考模型确实提高了。观察 β 对训练的影响（试 0.1/0.5/1.0）。讨论：DPO 会不会让模型"钻空子"（reward hacking 的 DPO 版）？
